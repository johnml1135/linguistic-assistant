# Variant form

> A lexical entry that is an alternate version of another entry — a dialectal, spelling, free, or
> irregularly-inflected variant — recorded as a reference from the variant to its main entry.

**LibLCM class:** `LexEntryRef` (RefType = variant)  ·  **FLEx UI:** "Variant of" / "Variants"
·  **Tier:** lexical  ·  **In MiniLcm:** partial

## What it is (linguistics)

A variant form is the same lexeme realized differently: *colour* vs *color* (spelling), *gray* vs
*grey* (dialectal), *isn't* vs *ain't* (free/social), or an irregular inflected form treated as a
listed variant (*went* as a variant of *go*). Unlike a [[complex-form]] — which is built *from* several
entries — a variant points at exactly **one** main entry, the form a reader should consult. FLEx's
default **variant types** include: dialectal variant, free variant, spelling variant, and irregularly
inflected (inflectional) variant; the list is user-extensible.

The point of recording variants (rather than minting independent entries) is that they share meaning
with the main entry and should resolve to it in lookup, parsing, and publication.

## How LibLCM models it

Like a [[complex-form]], a variant is a `LexEntryRef` — same class, distinguished by `RefType` flagged
as *variant*. It is owned in the variant entry's `LexEntry.EntryRefs`. Key fields:

- **`ComponentLexemes`** — for a variant, this holds the single main `LexEntry`/`LexSense` it varies
  from (FLEx labels it "Variant of"); `PrimaryLexemes` and `ShowComplexFormsIn` are *unused* for
  variants.
- **`VariantEntryTypes`** — reference sequence of `LexEntryType` (a `CmPossibility` subclass):
  dialectal, spelling, free, inflectional, etc. (`ComplexEntryTypes` is its sibling, unused here.)
- **`HideMinorEntry`** — `Integer` (default 0); controls whether the variant appears as a minor entry
  in the dictionary.

**`RefType` is a plain `Integer` enum on `LexEntryRef`** (0 = variant, 1 = complex form), *not* a typed
reference. It is only a discriminator telling the UI whether to render this ref as a variant or a
complex form; the actual type list lives in the separate `VariantEntryTypes` reference collection (each
item a `LexEntryType`). Inflectional variants additionally use `LexEntryInflType` (class 133, a
`LexEntryType` subclass carrying `InflFeats` — the inflection features the irregular form realizes — and
`Slots`/`GlossPrepend`/`GlossAppend`). Verified against `MasterLCModel.xml` (class 127 `LexEntryRef`,
class 133 `LexEntryInflType`).

## Hermit Crab mapping

Irregularly inflected variants matter most to HC: *went* cannot be generated by regular rules from
*go*, so it is supplied as a stored allomorph/listed form so the parser still maps it to the main
lexeme and its features. Spelling/dialectal variants are largely a lexicography/display concern, though
a spelling variant may need its own [[allomorph]] if it should parse.

## In our change-sets

```yaml
op: lexical.variant.create
entry: { lexeme_form: { seh: "imbgwa" } }
variant_type: dialectal
variant_of: "entry:imbwa"
rationale: "Coastal-dialect spelling of 'dog'; should resolve to main entry."
confidence: 0.7
provenance: { corpus: "luke", refs: ["LUK 16:21"] }
```

## QA & parallel relevance

[[parallel-translation-qa]] flags a vernacular token that is clearly a variant spelling of an existing
entry but parses as unknown — a "missing-variant" data gap. Consistency checks catch the inverse: two
independent entries that are really spelling variants of one lexeme (should be merged with a variant
link, see [[homograph-number]] for the split/merge decision).

## Pitfalls

- **Variant vs separate entry.** Promoting every dialect spelling to its own entry fragments the
  lexicon and breaks gloss/parse consistency.
- **Variant vs [[complex-form]].** Identical class; the `RefType` flag and a single-vs-many
  `ComponentLexemes` target are the discriminators.
- **Inflectional variant without features** parses but contributes no morphosyntax — a silent gap in
  [[interlinearization]].
